# 前端对接任务：注册支持邀请码（invite_code）落组

> 负责人：前端工程师乙（web/user-console）
> 对应后端变更：PR #182「注册落组」（已合并 main，已部署测试环境）
> 接口契约权威：`docs/full-api-design.md` 2.3 节、`docs/frontend-api-reference.md`

## 一、背景（你只需了解，不需实现后端逻辑）

后端注册接口 `POST /api/auth/register` 新增了一个**可选**字段 `invite_code`。用户注册成功后，后端会按邀请码把新用户落入对应分组；不传或邀请码无效时落入默认组。**落组是后端行为，前端只负责把邀请码传过去。**

## 二、接口契约（唯一需要对接的变更）

`POST /api/auth/register` 请求体在原有字段基础上**新增一个可选字段**：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| invite_code | string | **否** | 组邀请码。用户填了就传，没填就不传（或传空字符串） |

请求体示例：

```json
{
  "username": "alice01",
  "phone": "13800001234",
  "email": "alice@example.com",
  "password": "Alice@123",
  "phone_code": "123456",
  "email_code": "654321",
  "invite_code": "ABC12345"
}
```

**响应结构完全不变**（仍是 201 + `access_token`/`refresh_token`/`expires_in`/`user`），无需改动响应处理。

## 三、前端需要做的改动（用户控制台注册页）

1. **类型**：`web/user-console/src/types/auth.ts` 中注册请求类型新增可选字段 `invite_code?: string`。
2. **API 封装**：`web/user-console/src/api/auth.ts` 的 `register` 调用透传 `invite_code`（仅当用户填写时带上该字段）。
3. **注册页**：`web/user-console/src/views/auth/RegisterView.vue` 增加一个**选填**的「邀请码」输入框：
   - 标签建议：`邀请码（选填）`；占位符如 `如有邀请码请填写，没有可留空`；
   - 不做必填校验、不做格式/长度的前端校验（邀请码规则由后端判定）；
   - 用户留空时，提交的 body **不要带 `invite_code` 字段，或传空字符串**（两者后端等价处理）。

## 四、重要：不需要处理的事项（避免过度实现）

- ❌ **不要**对邀请码做前端有效性校验或调接口预校验——后端没有「校验邀请码」的独立接口。
- ❌ **不要**为「邀请码无效」写错误提示分支。后端采用「方案 A」：**邀请码无效/过期/已满不会报错**，会静默把用户落入默认组，注册照常成功（仍返回 201）。前端按正常注册成功流程走即可。
- ❌ **不要**在前端展示用户「落入了哪个组」——注册响应不返回分组信息，本期不涉及。

## 五、验收口径

- 填了有效邀请码注册 → 注册成功（201），正常进入登录态。
- 填了无效邀请码注册 → **同样注册成功**（不报错）。
- 不填邀请码注册 → 与改动前行为一致。

> 后端落组结果的验证由测试工程师用 `tests/test_register_default_group.py` 覆盖，前端无需自测落组。

## 六、范围边界

- 本任务只动 **web/user-console** 注册相关的类型/接口封装/注册页。
- 管理后台的分组/邀请码管理（前端工程师甲，web/admin-console）已有独立功能，**不在本任务范围**。
- 后端逻辑、数据库、鉴权一律不动。
